### Organization / Team Administration (ORG)

Honest disposition of the ORG stories against the current codebase. The domain is essentially **unbuilt as a product surface**: there is no admin console route (`app/` has no org/roster/cohort/admin path), no org service (`lib/services/` has none), and no invite/seat/cohort/assignment/policy entities. What exists are two foundations that ORG *depends on* and points at:

- **Data-model stub** — `lib/store/entities.ts`: `Org { id, name, plan: "team", seatLimit }`, `User.orgId`, `User.plan`, and an `orgs` map in `lib/store/db.ts`.
- **Tenant-scoped RBAC + epistemic firewall** — `lib/domain/rbac.ts` (tested in `rbac.test.ts`): the 12-role matrix incl. `org_admin` / `billing_admin` / `instructor` / `team_learner` / `compliance_dpo`, separable capabilities (`org:read`, `org:manage_seats`, `org:manage_roles`, `billing:manage`, `tenant:provision`, `audit:read`), and the master firewall — **no human role holds `trust:write`**; only the system-verifier / SME dual-control path in `trust.ts` can set trust. `conflicts.ts` re-verifies disputed claims via the system verifier only.

Because none of these stories are wired to a real admin surface and none is *genuinely blocked* except SSO/SCIM, they are marked Partial (primitive/seam exists, product behavior missing) per the rules, with the one vendor-blocked story Deferred.
