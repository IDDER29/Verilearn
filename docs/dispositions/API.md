### Integrations, API, Webhooks, SSO & LTI (API)

This domain is the external/federated surface (public REST/GraphQL, API keys, OAuth, signed webhooks, SSO, SCIM, LTI, verify endpoint/widget, sandbox, versioning). The codebase has **no HTTP API surface**: `app/` contains no `route.ts` / `api/` handlers, and there is no webhook, OAuth, SAML/OIDC, SCIM, or LTI code anywhere in `web/`. What *is* built are the internal domain primitives any future API would sit on: fail-closed certificate verify (`lib/domain/certificates.ts`), the five-state trust ledger + coverage matrix (`lib/domain/trust.ts`, `lib/services/sources.ts`), the four honest signals (`lib/domain/signals.ts`), and the epistemic firewall in the capability layer (`lib/domain/rbac.ts`, `lib/domain/trust.ts`). Classifications reflect that reality.

| API-03 | ✅ Done | The programmatic public verify endpoint now exists: `GET /api/verify/[code]` (`app/api/verify/[code]/route.ts`), a real Next.js Route Handler — no vendor/external infra needed, since Route Handlers are a first-party framework primitive. It wraps the fail-closed `verifyCertificate()`/`publicVerify()` (`lib/services/certificates.ts`) — an unknown or revoked code never resolves valid — and is exempted from the session auth-gate (`proxy.ts`, `PUBLIC_PREFIXES`), verified via curl to be reachable with **no session cookie**. The response is a **signed envelope**: an HMAC-SHA256 (`X-VeriLearn-Signature` header) over the exact JSON body, verified by manual recomputation to match byte-for-byte. A malformed/empty code returns `400`; an infra fault in the lookup fails closed to **`503 unverifiable_try_again`** (a real `try/catch` around the store read, not a fabricated branch — this in-memory store has no genuine I/O failure mode today, but the fail-closed path is real code that runs if the lookup ever throws). The response is **PII-safe**: only `topicTitle`/`testScorePct`/`issuedAt` are returned on a valid cert — never the learner's id/name/email — verified in `lib/services/certificates.test.ts` (valid/revoked/unknown, no leaked PII, signature-stable). Remaining gap: **enumeration rate-limiting** needs shared/distributed limiter infrastructure this single-process app doesn't have. |
| API-04 | ✅ Done | A real REST transport now exists: `GET /api/topics/[id]/trust` (`app/api/topics/[id]/trust/route.ts`), backed by a new `trustReadModel()` (`lib/services/sources.ts`). It returns the five-state per-claim trust read (`Verified·execution/source`, `Sourced`, `Disputed`, `Unsupported`, `Interpretive` — no binary collapse) plus `verifiedPercent`. **`as_of` ledger versioning is real**: it replays only the event-sourced ledger's events recorded at-or-before the requested `?as_of=<epoch_ms>`, a genuine historical snapshot rather than an approximation — verified in `sources.test.ts` (an event after the cutoff is invisible to an earlier `as_of`, visible at/after its own timestamp). **`pipeline_incomplete` status** is real: a topic whose verification run hasn't finished (`status !== "ready"`) always reads `pipeline_incomplete`, never presenting an in-flight/failed topic as a finalized read. The route is a JSON API (`proxy.ts` `JSON_API_PREFIXES`) that returns a real `401` JSON body (not an HTML redirect) when unauthenticated — verified live via curl. **`trust:read` scope**: rather than a separate API-key/OAuth scope (deferred infra, API-01/02), the endpoint is scoped to the authenticated session + tenant-ownership check (`topic.ownerId === user.id`) — documented honestly as a session-based stand-in, not the eventual credential model. |
